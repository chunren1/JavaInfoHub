## 3. 数据库设计

### 3.1 tech_article 表（技术文章）

```sql
CREATE TABLE IF NOT EXISTS tech_article (
    id            BIGINT       AUTO_INCREMENT PRIMARY KEY COMMENT '主键',
    title         VARCHAR(500) NOT NULL                COMMENT '文章标题',
    summary       TEXT                                 COMMENT '摘要/描述',
    content_url   VARCHAR(1000) NOT NULL               COMMENT '原文链接',
    author        VARCHAR(200)                         COMMENT '作者/项目拥有者',
    tags          VARCHAR(500)                         COMMENT '标签（逗号分隔）',
    source        VARCHAR(50)  NOT NULL                COMMENT '来源：JUJIN/SEGMENTFAULT/GITHUB/OSCHINA',
    source_id     VARCHAR(200) NOT NULL                COMMENT '源站唯一ID',
    publish_time  DATETIME                             COMMENT '发布时间',
    view_count    INT          DEFAULT 0               COMMENT '浏览数',
    star_count    INT          DEFAULT 0               COMMENT '点赞/Star/Fork数',
    dedup_key     VARCHAR(64)  NOT NULL UNIQUE         COMMENT 'MD5 去重键：MD5(source + source_id)',
    crawl_time    DATETIME     DEFAULT CURRENT_TIMESTAMP COMMENT '首次抓取时间',
    update_time   DATETIME     DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',

    INDEX idx_source        (source),
    INDEX idx_publish_time  (publish_time),
    INDEX idx_crawl_time    (crawl_time),
    FULLTEXT INDEX ft_title_summary (title, summary)   -- MySQL 全文索引，支持中文搜索
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='技术文章聚合表';
```

### 3.3 去重键设计

```
dedup_key = MD5(source + ":" + source_id)
```

- 作用：同一来源的同一文章/职位不会重复入库
- INSERT IGNORE 兜底：即使内存去重漏了，DB UNIQUE 约束也能保证不重复
- 面试展示：这是生产级爬虫的标准做法

```java
    /**

     *
     * 适用场景：
     *  - 页面数据藏在 &lt;script&gt; 里的 JS 变量中（如 window.__INITIAL_STATE__）
     *  - 接口参数需要从 JS 中提取（如加密 token、签名参数）

     *
     * 面试话术："有些站点的数据不直接渲染在 HTML 中，而是放在 JS 变量里。
     *           我用这方法从 <script> 标签中提取数据，再用 JSON 解析。"
     */
    public static String jsVarExtractor(String html, String varPattern);

    /**
     * 【面试重点】从 JS 函数中提取参数值
     *
     * 示例：
     *   JS: fetch('/api/list', {headers: {'X-Token': 'abc123'}})
     *   调用: jsValueExtractor(html, "'X-Token'\\s*:\\s*'([^']+)'")
     *   返回: "abc123"
```

### 4.3 webmagic-core（爬虫核心）

#### 4.3.1 5 个 PageProcessor 技术详解

| Processor | 核心解析方式 | 关键技术点 | JD 覆盖 |
|-----------|-------------|-----------|---------|
| **JuejinPageProcessor** | POST JSON API + cursor 分页 | ① 构造 JSON Body 发送 POST 请求② 解析 JSON 响应③ cursor 参数实现无限滚动分页④ 设置 X-Requested-With / Content-Type 等 Ajax Header | JSON/Ajax、HTTP Headers |
| **SegmentfaultPageProcessor** | jsoup CSS 选择器 | ① `.news-item .news-title a` 提取标题② `.news-item .news-summary` 提取摘要③ `.news-item .author-name` 提取作者④ URL `?page=N` 翻页到 maxPage | CSS 选择器 |
| **GithubTrendingProcessor** | CSS + 正则 | ① CSS `.Box-row h2 a` 提取仓库名② **正则** `\d[\d,]*` 提取 star/fork 数字③ 正则 `built by.*` 提取语言④ UA 随机轮换避免限流 | CSS、正则、UA 轮换 |
| **ShixisengProcessor** | CSS + XPath 混合 | ① CSS `.intern-job-list .job-item` 提取列表② XPath `//div[@class='job-msg']/span[1]` 提取薪资③ XPath `//div[@class='job-msg']/span[2]` 提取城市④ 携带 Cookie 模拟已登录 | CSS、XPath、Cookie |
| **OsChinaProcessor** | 纯 XPath | ① `//div[@class='news-item']` 提取列表② `//a[@class='news-title']/text()` 提取标题③ XPath 位置谓词 `//span[position()=2]` 提取作者④ XPath 函数 `contains(@class, 'view')` 提取浏览数 | XPath（全覆盖） |


#### 4.3.3 DedupPipeline 设计

```java
@Component
public class DedupPipeline implements Pipeline {
    // 内存去重：ConcurrentHashMap 快速路径（避免每次查 DB）
    private final ConcurrentHashMap<String, Boolean> urlCache = new ConcurrentHashMap<>();

    // 最大缓存数量（防止内存溢出）
    private static final int MAX_CACHE_SIZE = 10_000;

    @Override
    public void process(ResultItems resultItems, Task task) {
        String dedupKey = resultItems.get("dedupKey");
        if (dedupKey == null) return;

        // 1. 内存快速去重
        if (urlCache.containsKey(dedupKey)) {
            log.debug("内存去重命中：{}", dedupKey);
            resultItems.setSkip(true); // 跳过后续 Pipeline
            return;
        }

        // 2. 缓存管理
        if (urlCache.size() >= MAX_CACHE_SIZE) {
            urlCache.clear(); // 简化策略：写满就清
        }
        urlCache.put(dedupKey, true);
    }
}
```

## 6. HTTP 状态码与反爬对抗

### 6.1 退避重试策略

在 `CrawlSchedulerService` 中实现基础版的重试逻辑：

```java
/**
 * 【JD 覆盖：HTTP 状态码处理】
 * 对于 429 Too Many Requests / 503 Service Unavailable 等反爬状态码，
 * 实现指数退避重试策略。
 */
private boolean crawlWithRetry(String source, int maxRetries) {
    for (int attempt = 1; attempt <= maxRetries; attempt++) {
        try {
            Spider spider = createSpider(source);
            spider.run();
            return true;
        } catch (Exception e) {
            // 判断是否为可重试的 HTTP 状态码
            if (isRetryableHttpError(e) && attempt < maxRetries) {
                long delaySeconds = (long) Math.pow(2, attempt);  // 指数退避: 2s → 4s → 8s
                log.warn("爬取 {} 遇到可重试错误，第 {} 次重试，等待 {} 秒", source, attempt, delaySeconds);
                SleepUtils.sleep(delaySeconds * 1000);
            } else {
                log.error("爬取 {} 失败，已达最大重试次数", source);
                return false;
            }
        }
    }
    return false;
}

/** 判断是否可重试的 HTTP 状态码 */
private boolean isRetryableHttpError(Exception e) {
    String msg = e.getMessage();
    return msg.contains("429")   // Too Many Requests — 限流
        || msg.contains("503")   // Service Unavailable — 服务暂不可用
        || msg.contains("502")   // Bad Gateway
        || msg.contains("504");  // Gateway Timeout
}
```

## 8. Pipeline 去重链

### 8.1 去重架构

```
    ┌──────────────────┐
    │   ResultItems    │ ← PageProcessor 产出的实体
    └────────┬─────────┘
             ↓
    ┌──────────────────────────┐
    │  DedupPipeline           │
    │                          │
    │  1. 计算 MD5 去重键      │
    │  2. 查内存 ConcurrentMap │ ← 快速路径 ~0.01ms
    │  3. 查 DB（INSERT IGNORE │ ← 持久化兜底
    │     的唯一索引也可去重） │
    │                          │
    │  命中 → resultItems      │
    │         .setSkip(true)   │ ← 跳过后续 Pipeline
    └────────┬─────────────────┘
             ↓ (非重复)
    ┌──────────────────────┐
    │  ArticlePersistPipeline│
    │  批量 INSERT IGNORE   │ ← DB 唯一索引二次保护
    └──────────────────────┘
```

### 8.2 面试话术

```
"去重是生产级爬虫的核心环节。我用了两层去重：
第一层是内存 ConcurrentHashMap 快速判断，性能高；
第二层是数据库 UNIQUE 索引，保证即使内存去重遗漏，
INSERT IGNORE 也不会插入重复数据。
去重键的设计是 MD5(source + source_id)，不是简单的 URL 去重，
因为同一个 URL 可能对应不同来源。"
```

## 12. 风险预案与 Fallback

### 12.1 各数据源风险评估

| 站点 | 风险 | 概率 | Fallback |
|------|------|------|----------|
| 掘金 API | API 接口变更 | 低 | 该接口是公开的推荐接口，较稳定；若变更可用浏览器抓包重新定位 |
| SegmentFault | 页面改版 | 中 | CSS 选择器需微调；可在面试前 1 天验证 |
| GitHub Trending | 限流（429） | 中 | UA 轮换 + 重试退避；面试时只抓一页 25 条 |
| 实习僧 | Cookie 失效或反爬升级 | **高** | 📌 备用方案 1：改为拉勾网部分公开列表；📌 备用方案 2：预置 mock 数据 |
| 开源中国 | 页面改版 | 低 | XPath 需微调；结构较稳定 |

### 12.2 实习僧 Fallback 方案

```
方案 A（推荐）：面试前 1 天验证实习僧是否可爬
  - 如果不行 → 替换为「拉勾教育」或「牛客网」的公开列表页
  - 提前准备好替换的 Processor

方案 B：预置数据兜底
  - schema.sql 中附带 init-data.sql
  - 面试前导入 100+ 条预置数据（涵盖所有来源）
  - 即使现场某个站点爬取失败，Demo 也有数据展示
  - 可以说："为了演示效果，我预置了一些历史数据，现在展示实时爬取"

方案 C：兜底 Processor
  - 写一个通用的 CSS 解析 Processor
  - 如果实习僧不行，现场改 CSS 选择器指向另一个站点
```

### 12.3 技术故障应急预案

| 故障 | 应对 |
|------|------|
| MySQL 连不上 | 检查 application.yml 中的数据库配置，确认 MySQL 服务已启动 |
| 端口 8080 被占用 | `server.port=8088` 或 `netstat -ano | findstr 8080` 杀掉进程 |
| Maven 依赖下载慢 | 提前 `mvn clean compile` 确保依赖已缓存到本地仓库 |
| 某个 Processor 报错 | 在 CrawlController 中加了 try-catch，单个站点失败不影响整体 |
